# Git Branch + Worktree 연습 플랜

## 목표
- 브랜치 기반 작업 흐름과 worktree 병렬 작업 흐름을 한 번에 익힌다.
- `main` 기준으로 생성, 전환, 푸시, 병합, 정리까지 전체 사이클을 연습한다.

## 1. Clone 후 기준 상태 확인
- 저장소를 클론한 뒤 현재 브랜치와 작업 상태를 확인한다.
- 시작 시점이 `origin/main`과 맞는지 점검한다.

## 2. 브랜치 생성 후 Push
- 새 브랜치를 만들고 첫 커밋을 원격으로 푸시한다.
- 로컬 브랜치와 원격 브랜치 추적 관계를 확인한다.

## 3. 브랜치 전환(switch) 연습
- 브랜치를 바꿨을 때 현재 작업 디렉터리 기준이 어떻게 바뀌는지 확인한다.
- "내가 지금 어느 브랜치에 서 있는지"를 항상 확인하는 습관을 만든다.

## 4. 전환한 브랜치에서 수정 후 Push
- 파일 수정 → 커밋 → 푸시를 진행한다.
- GitHub에서 해당 브랜치를 선택했을 때 변경 내용이 보이는지 확인한다.
- `main`에서 바로 안 보이는 것이 정상임을 이해한다(merge 전).

## 5. 브랜치 제거
- 실습 브랜치를 로컬/원격 각각 정리한다.
- 로컬 삭제와 원격 삭제가 별도 동작임을 확인한다.

## 6. worktree + branch 세팅
- 새 worktree 디렉터리를 만들면서 연결 브랜치를 지정한다.
- 같은 저장소를 서로 다른 폴더에서 동시에 다루는 구조를 확인한다.

## 7. worktree에서 파일 생성 및 커밋
- worktree 쪽 브랜치에서 파일을 추가하고 커밋한다.
- 메인 작업 폴더에 영향 없이 브랜치 이력만 바뀌는지 확인한다.

## 8. main에서 별도 파일 생성 및 커밋
- 원래 메인 작업 폴더(`main`)에서 다른 파일을 추가하고 커밋한다.
- 병렬로 쌓인 커밋 이력이 어떻게 갈라지는지 확인한다.

## 9. 최종 merge
- `main`에서 worktree 브랜치를 병합한다.
- 충돌이 나면 수동으로 해결하고 병합 커밋을 완료한다.
- 최종 결과를 로컬 로그와 원격(GitHub)에서 함께 확인한다.

## 완료 체크리스트
- 브랜치 생성/전환/푸시를 스스로 수행했다.
- 원격 브랜치와 로컬 브랜치의 차이를 설명할 수 있다.
- worktree가 "새 저장소 clone"이 아니라 "같은 저장소의 추가 작업 디렉터리"임을 설명할 수 있다.
- merge와 conflict 해결을 한 번 이상 직접 수행했다.
